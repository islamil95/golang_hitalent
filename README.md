# API организационной структуры

REST API для управления подразделениями (departments) и сотрудниками (employees) с древовидной структурой. Реализовано на Go с использованием **net/http**, **GORM**, **PostgreSQL** и миграций **goose**.

## Запуск

### Docker Compose (рекомендуется)

```bash
docker-compose up --build
```

API будет доступно по адресу: **http://localhost:8080**

База данных PostgreSQL в контейнере доступна на хосте по порту **5433** (чтобы не конфликтовать с локальным PostgreSQL на 5432). Логин/пароль: `postgres`/`postgres`, БД: `org_structure`.

Миграции выполняются автоматически при старте приложения.

**Если при запуске ошибка «port is already allocated» для 5432** — значит порт занят (например, локальный PostgreSQL). В нашем `docker-compose` БД уже проброшена на **5433**, перезапустите: `docker compose up --build`.

**GET /departments/1 возвращает 404?** Это нормально, пока в БД нет подразделений. Сначала создайте отдел: `POST /departments/` с телом `{"name": "Backend"}`.

 
## Описание API

### 1. Создать подразделение

**POST** `/departments/`

Тело запроса:
```json
{
  "name": "Backend",
  "parent_id": null
}
```

`parent_id` — необязательный (корневое подразделение, если не указан).

### 2. Создать сотрудника в подразделении

**POST** `/departments/{id}/employees/`

Тело запроса:
```json
{
  "full_name": "Иван Иванов",
  "position": "Разработчик",
  "hired_at": "2024-01-15"
}
```

`hired_at` — необязательный (дата в формате `YYYY-MM-DD`).

### 3. Получить подразделение (детали, сотрудники, поддерево)

**GET** `/departments/{id}?depth=1&include_employees=true`

Параметры запроса:
- `depth` — глубина вложенных подразделений (по умолчанию 1, максимум 5).
- `include_employees` — включать ли сотрудников (по умолчанию `true`).
- `sort_employees_by_name` — при `true` сортировка сотрудников по `full_name`, иначе по `created_at`.

Ответ: объект подразделения, массив `employees`, массив `children` (рекурсивно до `depth`).

### 4. Обновить подразделение

**PATCH** `/departments/{id}`

Тело запроса (все поля необязательные):
```json
{
  "name": "Новое название",
  "parent_id": 2
}
```

Нельзя сделать подразделение родителем самого себя и нельзя создать цикл в дереве.

### 5. Удалить подразделение

**DELETE** `/departments/{id}?mode=cascade`

Параметры запроса:
- `mode` — `cascade` (по умолчанию) или `reassign`.
- При `mode=reassign` обязателен параметр `reassign_to_department_id` — ID подразделения, в которое переводятся сотрудники и дочерние подразделения.

Ответ: **204 No Content**.

## Ограничения и валидация

- Название подразделения: не пустое, длина 1–200 символов, пробелы по краям обрезаются. В пределах одного родителя имена уникальны.
- ФИО и должность сотрудника: не пустые, длина 1–200 символов.
- Создание сотрудника в несуществующем подразделении — **404**.
- Перемещение подразделения в себя или в своё поддерево — **409 Conflict** / **400**.

## Структура проекта

```
.
├── cmd/api/           # Точка входа приложения
├── internal/
│   ├── config/        # Конфигурация
│   ├── handler/      # HTTP-обработчики и роутер
│   ├── middleware/   # Логирование, recovery
│   ├── model/        # Модели GORM (Department, Employee)
│   ├── repository/   # Работа с БД
│   └── service/      # Бизнес-логика и валидация
├── migrations/       # SQL-миграции goose
├── Dockerfile
├── docker-compose.yml
└── README.md
```

## Технологии

- **Go 1.22**
- **net/http** — HTTP-сервер и роутинг
- **GORM** — ORM для PostgreSQL
- **goose** — миграции БД
- **Docker / docker-compose** — контейнеризация и запуск окружения
